# Proposal: Name Output Files After Their Definition

**Date:** 2026-04-13
**Related:** functionality-improvements.md suggestion #2

---

## Problem

Every result file written to the output directory currently has a UUID as its name:

```
output/3f2a91b0-4c2d-4e1a-8b3f-9e74c1cd67e1
output/7c91ad22-11b4-4f8a-a02c-3b6e2d4f8901
```

A user cannot tell from the directory listing which result belongs to which definition.
They must consult the summary table printed to stdout, copy the UUID, and then open the
right file. This makes the output directory nearly unusable as a handoff point for downstream
tools or manual inspection.

---

## Goal

Retained definitions (those written by the user) produce output files named after the
definition, with a `.txt` extension:

```
output/Added.txt
output/Removed.txt
```

Transient definitions (internally generated intermediate results) keep their UUID names
since they are implementation details and their `sdRetain` flag is already `False`.

---

## Filename safety analysis

### User-defined identifiers

The lexer character class for identifiers is defined in `src/SetLanguage.x`:

```alex
$ident = [a-zA-Z0-9\-\_]
```

Every character in this set is safe on all major filesystems:

| Character class | Linux | macOS | Windows (NTFS) |
|----------------|-------|-------|---------------|
| `a-z`, `A-Z` | ✓ | ✓ | ✓ |
| `0-9` | ✓ | ✓ | ✓ |
| `-` (hyphen) | ✓ | ✓ | ✓ |
| `_` (underscore) | ✓ | ✓ | ✓ |

No escaping or sanitisation of identifier characters is needed.

Adding `.txt` as the extension is also universally safe and clearly signals the file
type to downstream tools.

### Internally generated identifiers (transient)

Transient identifiers are generated by `integerToId` in `src/ExpressionConversion.hs`:

```haskell
integerToId :: Integer -> Identifier
integerToId = T.pack . show
```

These are decimal integers as text strings (`"0"`, `"1"`, `"42"`, etc.). They are valid
filenames on all platforms. However, since transient definitions keep UUID names, this is
moot in practice.

### The `standard/example.setdown` edge case

The existing example contains `1 : "a-1.out"` — a purely numeric identifier. The output
file would be `1.txt`, which is a valid filename everywhere. No special handling needed.

### Collision with the output directory itself

If a user names a definition `output` (the default output directory name), the result file
would be `output/output.txt` — a file inside the directory, not a name conflict with the
directory itself. No special handling needed.

## Deduplication and shared underlying files

`eliminateDuplicates` (in `src/DuplicateElimination.hs`) merges definitions with identical
expressions so they share a single computed result file. This means multiple retained
definitions can legally point to the same UUID file at the time results are returned from
`runSimpleDefinitions`.

Example:

```setdown
Shared:    "app-a.txt" /\ "app-b.txt"
AlsoShared: "app-a.txt" /\ "app-b.txt"
```

Both `Shared` and `AlsoShared` will map to the same UUID file. If we **rename** the source
file, the second rename attempt will fail because the source no longer exists. We must
**copy** instead, writing the same content to both `Shared.txt` and `AlsoShared.txt`.

After all copies are complete, UUID files that were the source of at least one retained
definition copy can be deleted (they are now redundant). UUID files belonging to transient
definitions should be left in place, as they may be inspected via `--show-transient`.

## Implementation plan

All changes are confined to `src/Main.hs`. `PerformOperations.hs` and `ExternalSort.hs`
are untouched.

### Step 1 — write a `publishResults` function

After `runSimpleDefinitions` returns `computedFiles :: [(SimpleDefinition, FilePath)]`,
call a new function that copies retained results to named paths and returns an updated list:

```haskell
publishResults :: Context -> [(SimpleDefinition, FilePath)] -> IO [(SimpleDefinition, FilePath)]
publishResults ctx results = do
   updated <- mapM publish results
   cleanupOrphanedUUIDs ctx results updated
   return updated
  where
   publish (sd, src)
      | sdRetain sd = do
            let dest = cOutputDir ctx </> TL.unpack (sdId sd) ++ ".txt"
            copyFile src dest
            return (sd, dest)
      | otherwise = return (sd, src)
```

`copyFile` is available from `System.Directory` (already imported).

### Step 2 — clean up orphaned UUID files

After all copies are done, delete UUID source files that are no longer the canonical path
for any result (i.e. files that were the source of at least one retained copy and are not
themselves the destination of anything):

```haskell
cleanupOrphanedUUIDs :: Context
                     -> [(SimpleDefinition, FilePath)]  -- original (UUID paths)
                     -> [(SimpleDefinition, FilePath)]  -- updated (named paths)
                     -> IO ()
cleanupOrphanedUUIDs _ before after = do
   let retainedSources = S.fromList [src | (sd, src) <- before, sdRetain sd]
   let newPaths        = S.fromList (fmap snd after)
   let orphans         = retainedSources S.\\ newPaths
   mapM_ removeFile (S.toList orphans)
```

`removeFile` is from `System.Directory`.

### Step 3 — wire into `main`

Replace the existing:

```haskell
computedFiles <- runSimpleDefinitions context simpleSetData sortedFiles
printComputedResults opts computedFiles
```

with:

```haskell
computedFiles  <- runSimpleDefinitions context simpleSetData sortedFiles
publishedFiles <- publishResults context computedFiles
printComputedResults opts publishedFiles
```

The summary table in `printComputedResults` then naturally shows the named paths.

### New imports required in `Main.hs`

```haskell
import System.Directory (copyFile, removeFile, ...)
```

`System.Directory` is already imported; `copyFile` and `removeFile` just need to be added
to the import list.

## Behavioural changes

| Scenario | Before | After |
|----------|--------|-------|
| Normal run | `output/<uuid>` | `output/MyDef.txt` |
| Re-run | Accumulates new UUID files | Overwrites named file in place |
| Two defs with identical expressions | Both point to same UUID | Each gets its own named copy |
| Transient defs (`--show-transient`) | UUID files in output | UUID files unchanged |
| Summary table paths | Long UUID paths | Short readable names |

---

## Out of scope for this change

- Renaming sorted input files (e.g. `api-v1.txt.1.split.sorted`) — these are internal
  artefacts of the external sort; a separate cleanup proposal covers this.
- Custom output filenames per definition (e.g. `Output "my-file.txt": expr`) — would
  require a language change.
- Atomic writes (write to temp then rename) — worthwhile for large files but a separate
  concern.
